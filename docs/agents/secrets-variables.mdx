## Overview

Preset agents can reference secrets and variables using Tracecat expression syntax.
Include `${{ SECRETS.<name>.<key> }}` or `${{ VARS.<name>.<key> }}` in a preset agent's instructions so the agent passes them as tool arguments at runtime.

<Note>
**Security model**: Preset agent tool calls go through a secure server-side proxy.
The LLM only sees the raw expression placeholder (e.g. `${{ SECRETS.threatintel.API_KEY }}`) — actual secret values are injected server-side at the tool execution layer after the model responds, so the model never has access to your credentials.
</Note>

<Warning>
**Do not use `${{ SECRETS... }}` or `${{ VARS... }}` in `ai.action` or `ai.agent` instructions or prompts.**
These actions evaluate all expressions before the LLM call. Secret expressions will resolve to `None` (breaking your tool calls), and variable expressions will be expanded inline, exposing their values directly to the model. Use `ai.preset_agent` for secure secret and variable injection.
</Warning>

## Secrets

Use `${{ SECRETS.<secret_name>.<key> }}` to reference a secret stored in your workspace credentials.

<Prompt
  description="Sample: pass API key to HTTP requests"
  icon="lock"
  actions={["copy"]}
>
You are a security triage agent.

When you make HTTP requests to the threat intel API, pass `${{ SECRETS.threatintel.API_KEY }}` in the Authorization header as a Bearer token.

Do not print or return secret values in your final answer.
</Prompt>

In a workflow action this looks like:

```yaml
- ref: triage_alert
  action: ai.preset_agent
  args:
    preset: security-triage
    user_prompt: |
      Look up the reputation of this IP address:

      ${{ TRIGGER.source_ip }}
```

Where the `security-triage` preset instructions contain:

```text
You are a security triage agent.

When you make HTTP requests to the threat intel API, pass
`${{ SECRETS.threatintel.API_KEY }}` in the Authorization header
as a Bearer token.

Do not print or return secret values in your final answer.
```

## Variables

Use `${{ VARS.<variable_name>.<key> }}` for non-sensitive configuration such as base URLs, project IDs, or channel names.

<Prompt
  description="Sample: pass base URL and project config via variables"
  icon="variable"
  actions={["copy"]}
>
You are a security triage agent.

When you call the threat intel API, use `${{ VARS.threatintel.base_url }}` as the base URL.
When you file tickets, use `${{ VARS.ticketing.project_id }}` as the project ID and `${{ VARS.ticketing.base_url }}` as the base URL.
</Prompt>

In a workflow action:

```yaml
- ref: triage_alert
  action: ai.preset_agent
  args:
    preset: security-triage
    user_prompt: |
      Investigate this alert and file a ticket with your findings:

      ${{ TRIGGER.alert }}
```

Where the `security-triage` preset instructions contain:

```text
You are a security triage agent.

When you call the threat intel API, use `${{ VARS.threatintel.base_url }}`
as the base URL.
When you file tickets, use `${{ VARS.ticketing.project_id }}` as the
project ID and `${{ VARS.ticketing.base_url }}` as the base URL.
```

## Combining secrets and variables

Use secrets for credentials and variables for everything else.

<Prompt
  description="Sample: combined secrets and variables for alert triage"

Credentials:
- Threat intel API key: `${{ SECRETS.threatintel.API_KEY }}`
- Ticketing API key: `${{ SECRETS.ticketing.API_KEY }}`

Configuration:
- Threat intel base URL: `${{ VARS.threatintel.base_url }}`
- Ticketing base URL: `${{ VARS.ticketing.base_url }}`
- Ticketing project ID: `${{ VARS.ticketing.project_id }}`

Steps:
1. Look up the source IP using the threat intel API.
2. File a ticket with your findings in the ticketing system.

```yaml
- ref: investigate_and_respond

    max_tool_calls: 10
```

Where the `security-triage` preset instructions contain:

```text

and file a ticket.

- Threat intel API key: `${{ SECRETS.threatintel.API_KEY }}`
```

## OAuth secrets

OAuth tokens follow a different naming convention. The secret name is `<provider_id>_oauth` and the key is the provider ID in uppercase plus `_USER_TOKEN` or `_SERVICE_TOKEN`.

```yaml
# authorization_code grant (delegated user access)
${{ SECRETS.<provider_id>_oauth.<PROVIDER_ID_UPPER>_USER_TOKEN }}

# client_credentials grant (service-to-service)
${{ SECRETS.<provider_id>_oauth.<PROVIDER_ID_UPPER>_SERVICE_TOKEN }}
```

<Prompt
  description="Sample: pass OAuth token to HTTP requests"
  icon="key"
  actions={["copy"]}
>
You are a security triage agent.

When you make HTTP requests to the Microsoft Sentinel API, pass `${{ SECRETS.microsoft_sentinel_oauth.MICROSOFT_SENTINEL_USER_TOKEN }}` in the Authorization header as a Bearer token.

Do not print or return secret values in your final answer.
</Prompt>

In a workflow action:

```yaml
- ref: query_sentinel
  action: ai.preset_agent
  args:
    preset: sentinel-analyst
    user_prompt: |
      Query Microsoft Sentinel for recent incidents related to this IP:

      ${{ TRIGGER.source_ip }}
```

Where the `sentinel-analyst` preset instructions contain:

```text
You are a security triage agent.

When you make HTTP requests to the Microsoft Sentinel API, pass
`${{ SECRETS.microsoft_sentinel_oauth.MICROSOFT_SENTINEL_USER_TOKEN }}`
in the Authorization header as a Bearer token.

Do not print or return secret values in your final answer.
```

## Best practices

- **Only use `ai.preset_agent` for secrets and variables.** `ai.action` and `ai.agent` evaluate expressions before the LLM call, which breaks secret injection and exposes variable values to the model.
- **Never ask the agent to log or return secrets.** Add "Do not print or return secret values in your final answer" to your preset instructions.
- **Use variables for non-sensitive config.** Base URLs, project keys, and channel names belong in variables, not secrets.
- **One secret per integration.** Keep credentials grouped by service (e.g. a `ticketing` secret with `API_KEY` and `EMAIL` keys).
