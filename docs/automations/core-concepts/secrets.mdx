## Overview

Secrets store sensitive values that your workflows, tools, and agents can reference with expressions such as `${{ SECRETS.github.GITHUB_TOKEN }}`.

Use secrets for API keys, passwords, certificates, private keys, and other credentials that should not be embedded directly in workflow definitions or prompts.

## Secret types

Tracecat supports these workspace secret types:

- `custom`: arbitrary key-value credentials
- `ssh-key`: a single SSH private key
- `mtls`: a TLS client certificate and private key
- `ca-cert`: a CA certificate bundle

### Custom credentials

Custom credentials are the most flexible option. A single secret can hold one or more key-value pairs.

Use them for values such as:

- API keys and bearer tokens
- usernames and passwords
- hostnames or account IDs that belong with a credential
- provider-specific fields such as `AWS_ROLE_ARN`

Example:

<CodeGroup>
```yaml Secret shape
name: github
type: custom
environment: default
keys:
  - key: GITHUB_TOKEN
    value: ghp_xxxxx
```

```yaml Expression
headers:
  Authorization: Bearer ${{ SECRETS.github.GITHUB_TOKEN }}
```
</CodeGroup>

### TLS and mTLS secrets

For certificate-based authentication, Tracecat supports fixed key names for the typed secret formats:

- `mtls` uses `TLS_CERTIFICATE` and `TLS_PRIVATE_KEY`
- `ca-cert` uses `CA_CERTIFICATE`
- `ssh-key` uses `PRIVATE_KEY`

Tracecat validates these typed secrets as PEM-encoded material when you create or update them.

<CodeGroup>
```yaml mTLS secret
name: mtls
type: mtls
keys:
  - key: TLS_CERTIFICATE
    value: |
      -----BEGIN CERTIFICATE-----
      ...
      -----END CERTIFICATE-----
  - key: TLS_PRIVATE_KEY
    value: |
      -----BEGIN PRIVATE KEY-----
      ...
      -----END PRIVATE KEY-----
```

```yaml CA certificate secret
name: ca_cert
type: ca-cert
keys:
  - key: CA_CERTIFICATE
    value: |
      -----BEGIN CERTIFICATE-----
      ...
      -----END CERTIFICATE-----
```
</CodeGroup>

## AWS credentials

AWS integrations can use standard custom credentials, and the UI also supports an AssumeRole-style setup for secrets that include `AWS_ROLE_ARN`.

When that pattern is used, Tracecat exposes:

- a Tracecat AWS account ID
- a dedicated Tracecat principal ARN
- a workspace-scoped External ID

You create a role in the target AWS account, trust the Tracecat principal, require the External ID, and then store the target role ARN in your secret.

Example secret:

<CodeGroup>
```yaml Secret shape
name: aws
type: custom
keys:
  - key: AWS_ROLE_ARN
    value: arn:aws:iam::123456789012:role/tracecat-access
```

```yaml Expression
role_arn: ${{ SECRETS.aws.AWS_ROLE_ARN }}
```
</CodeGroup>

If your tool or action expects different AWS fields, you can also store them as normal custom key-value credentials.

## Secret environments

Every secret belongs to a workflow environment. If you do not set one, Tracecat uses `default`.

Workflow environments let you keep the same secret name while changing the actual credential by environment. The workflow or action environment determines which secret is resolved.

This is useful when you need the same automation logic to run against different targets, such as:

- different CrowdStrike tenants for `prod`, `staging`, or `lab`
- multiple Slack apps for separate workspaces or business units
- separate sandbox and production API credentials for the same vendor

Example:

<CodeGroup>
```yaml Secrets
- name: slack
  environment: prod
  keys:
    - key: SLACK_BOT_TOKEN
      value: xoxb-prod

- name: slack
  environment: sandbox
  keys:
    - key: SLACK_BOT_TOKEN
      value: xoxb-sandbox
```

```yaml Workflow
config:
  environment: sandbox
```
</CodeGroup>

In that workflow, `${{ SECRETS.slack.SLACK_BOT_TOKEN }}` resolves to the `sandbox` value.

## Using secrets in expressions

Reference secrets with:

```yaml
${{ SECRETS.<secret_name>.<key> }}
```

Examples:

<CodeGroup>
```yaml HTTP header
headers:
  Authorization: Bearer ${{ SECRETS.crowdstrike.FALCON_CLIENT_SECRET }}
```

```yaml Basic auth
auth:
  username: ${{ SECRETS.jira.JIRA_USEREMAIL }}
  password: ${{ SECRETS.jira.JIRA_API_TOKEN }}
```
</CodeGroup>

## How to use with agents

Agents support secret expressions in tool arguments and integration configuration. Tracecat resolves those expressions inside Tracecat before a tool call executes.

The important security property is that secret expressions are not meant to be copied into model prompts as raw values. Tracecat resolves them only where tool execution needs them, and secrets are pre-resolved on the host rather than looked up inside the untrusted runtime.

Use prompts that tell the agent to reference secrets by expression name instead of asking it to print or reveal them.

Recommended prompting strategy:

<CodeGroup>
```text Good
When you call tools, use `${{ SECRETS.slack.SLACK_BOT_TOKEN }}` for Slack authentication and `${{ SECRETS.jira.JIRA_API_TOKEN }}` for Jira authentication. Do not print secret values in your response.
```

```text Better
Use the Slack and Jira credentials through Tracecat expressions only. Keep credentials inside tool arguments, never in the final answer, and never ask the user to paste the values.
```
</CodeGroup>

For MCP-based agent tools, Tracecat also resolves secret and variable expressions in `stdio` environment settings before launching the integration.
