## Overview

Secrets store sensitive values for workflows, tools, and agents, and you reference them with expressions such as `${{ SECRETS.github.GITHUB_TOKEN }}`.

## Using secrets in expressions

Access a secret in expressions with:

```yaml
${{ SECRETS.<secret_name>.<key> }}
```

## Secret types

Tracecat supports these workspace secret types:

- Custom (`custom`): arbitrary key-value credentials
- SSH key (`ssh-key`): a single SSH private key
- mTLS (`mtls`): a TLS client certificate and private key
- CA certificate (`ca-cert`): a CA certificate bundle

### Custom credentials

Custom credentials are key-value credentials that you can use for API keys, tokens, usernames, passwords, or provider-specific fields such as `AWS_ROLE_ARN`.

### TLS and mTLS secrets

For certificate-based authentication, Tracecat uses fixed key names:

- mTLS secrets use `TLS_CERTIFICATE` and `TLS_PRIVATE_KEY`
- CA certificate secrets use `CA_CERTIFICATE`
- SSH key secrets use `PRIVATE_KEY`

## AWS credentials

`tools.aws_boto3` uses a single secret named `aws`, and its keys are optional because the integration supports multiple auth flows.

Supported keys:

- AWS access key ID: `AWS_ACCESS_KEY_ID`
- AWS secret access key: `AWS_SECRET_ACCESS_KEY`
- AWS session token: `AWS_SESSION_TOKEN`
- AWS region: `AWS_REGION`
- AWS profile: `AWS_PROFILE`
- AWS role ARN: `AWS_ROLE_ARN`

The boto flow checks credentials in this order:

1. `AWS_ROLE_ARN`
2. `AWS_PROFILE`
3. `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` + `AWS_SESSION_TOKEN`
4. `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY`

If `AWS_ROLE_ARN` is present, Tracecat uses STS `AssumeRole` and injects `TRACECAT_AWS_EXTERNAL_ID` at runtime.

## Secret environments

Every secret belongs to a Workflow environment, and if you do not set one, Tracecat uses `default`.
Environments let you keep the same secret name and change the credential by target, such as:

- Different CrowdStrike tenants for `prod`, `staging`, or `lab`
- Multiple Slack apps for separate workspaces or business units
- Separate sandbox and production API credentials for the same vendor

## Access secrets in agents

Agents can use secret expressions in tool arguments and integration configuration.
This is secure because Tracecat resolves secrets during tool execution, and the raw secret values are not sent to the LLM provider.

<CodeGroup>
```text
When you call Slack tools, use `${{ SECRETS.slack.SLACK_BOT_TOKEN }}`.
When you call Jira tools, use `${{ SECRETS.jira.JIRA_API_TOKEN }}`.
Do not print secret values in the final answer.
```
</CodeGroup>

This also applies to MCP `stdio` environment settings.
