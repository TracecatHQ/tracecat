## Overview

Variables store reusable non-secret values for workflows, tools, and agents.

## Using variables in expressions

Access a variable in expressions with:

```yaml
${{ VARS.<variable_name>.<key> }}
```

## Variable fields

Each variable has these fields:

- Name: a lowercase snake case name such as `jira` or `api_config`
- Values: one or more key-value pairs
- Environment: the Workflow environment where the variable applies

## When to use variables

Use variables for non-sensitive values such as:

- Base URLs
- Project IDs
- Queue names
- Routing rules
- Tenant-specific defaults

If the value is sensitive, use a [secret](/automations/core-concepts/secrets) instead.

## Variable environments

Every variable belongs to a Workflow environment, and if you do not set one, Tracecat uses `default`.
Environments let you keep the same variable name and change the value by target.

## Access variables in agents

Agents can use variable expressions in tool arguments and integration configuration.
Tracecat resolves variables during tool execution, and the values are not fetched directly by the LLM provider.

<CodeGroup>
```text
When you call Jira tools, use `${{ VARS.jira.base_url }}` as the base URL and `${{ VARS.jira.project_key }}` as the default project key.
```
</CodeGroup>

This also applies to MCP `stdio` environment settings.
